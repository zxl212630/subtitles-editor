# 视频播放、Seek 与拖拽预览性能优化方案

本文档记录了针对本项目视频播放、Seek 定位以及拖拽实时预览（Scrubbing）场景的深度性能优化分析报告与阶段化演进方案。

---

## 一、 当前架构及实现简述

当前项目的视频处理流水线由以下核心组件构成：
1. **`MediaPlayer`**：多媒体播放控制器。通过 `QTimer`（每 16ms 触发一次）在主线程驱动音视频同步逻辑。
2. **`FFmpegDecoder`**（主解码线程）：用于正常播放。包含独立的音视频解码队列，解码出的 RGBA 数据缓存在 `videoQueue_` 中，PCM 数据缓存在 `audioQueue_` 中。
3. **`SeekDecoder`**（预览解码线程）：专门负责拖拽预览。独立于主解码器，支持高频 Seek，在后台线程解码视频并缩放至显示区域大小。
4. **`SoftwareVideoRenderer`**：基于 CPU 的视频画面及字幕渲染器。通过 `QPainter` 在主线程将 RGBA 数据绘制成 `QImage` 并渲染到界面上。

---

## 二、 关键瓶颈分析

根据代码审计，当前项目在**播放同步、单次 Seek、高频拖拽预览、视频画面渲染**四个维度存在明显的性能瓶颈：

### 1. 播放与同步（Playback & Audio/Video Sync）
* **基于系统时钟的音视频同步精度缺陷**：在 `MediaPlayer::onPlaybackTimer` 中，`audioClock` 的计算直接依赖于 `playbackElapsedTimer_.elapsed()`（即基于系统 CPU 时钟计算出的播放时间轴），并没有参考实际已经被声卡消费掉的音频采样数据。在系统卡顿或线程阻塞时，容易导致音视频不同步（A/V Drift）。
* **QTimer 主线程驱动播放循环**：播放主循环跑在主线程的 `QTimer`（16ms 周期）中。如果 UI 线程在执行繁重的操作，主线程卡顿将直接导致播放卡顿、跳帧，甚至音频爆音。
* **主线程同步写入音频可能阻塞**：主线程在定时器回调中调用 `audioOutput_->write()` 写入 PCM 数据。如果声卡缓冲区已满，可能会造成主线程短暂卡死。

### 2. 单次定位（Single Seek）
* **双通道解码器并行造成的内存与 I/O 开销**：在 `MediaPlayer::load` 时，主线程会并发调用 `FFmpegDecoder` 和 `SeekDecoder` 的 `open()` 函数。这引起磁盘 I/O 竞争并且占用双倍内存，对于 4K 高分辨率视频极其吃力。
* **单次 Seek 触发的双重解码竞争**：当用户在时间轴进行单次点击（常规 Seek，非拖动）时，`MediaPlayer::seek` 内部会同时向 `seekDecoder_` 和 `decoder_` 发送 seek 请求，引起不必要的算力浪费。

### 3. 拖拽实时预览（Drag-Seek / Scrubbing）
* **高频 Seek 中断导致的“拖拽黑屏/卡死”**：8ms 的合并延迟窗口极小，对于视频 Seek 追帧而言，单帧解码耗时往往在 20ms - 80ms。高频的 `requestSeek` 导致后台 `SeekDecoder` 频繁被中断并重新 Seek，将大量的 CPU 算力浪费在无效的“中断 -> 清理 -> 重新 Seek”循环中，导致拖动时画面长时间全黑或定格。
* **向左拖拽（倒退拖动）时的严重卡顿**：向右（向前）拖拽时，`SeekDecoder` 的智能 seek 判定如果差值在 2 秒内，则不调用 `avformat_seek_file`，实现顺序解码。但向左（后退）拖拽时，差值为负，必然每次都触发 `avformat_seek_file` 定位至上一个关键帧（I 帧），然后再向后解码（追帧）。这导致后退拖拽非常卡顿。

### 4. 视频渲染性能（Video Rendering）
* **纯 CPU 图像拷贝与绘制**：`SoftwareVideoRenderer` 继承自 `QWidget`，使用 `QPainter` 在主线程中把 `QImage` 画到屏幕上。把大内存 RGBA 图像从后台拷贝到主线程，再由 CPU 进行缩放绘制，在 1080p 甚至 4K 下会大量消耗 CPU 资源并拖慢 UI 线程的交互响应。

## 三、 阶段化优化演进路线

### 阶段 1：拖拽实时预览深度优化（优先实施）
1. **优化 Seek 合并延迟**：
   * 将高频拖拽 Seek 的合并延迟时间（Coalesce Window）由 8ms 调整至更合理的 30ms - 40ms。
   * 引入任务调度策略：若解码线程正忙，舍弃队列中积压的旧请求，优先确保当前正在进行的解码能够完成并渲染，保证拖拽帧率输出（Scrubbing Framerate >= 15fps）。
2. **极速拖拽模式（Keyframe Scrubbing）**：
   * 在高速拖拽或向左倒退拖拽时，自动降低解码精度，**仅解码并显示关键帧（I-Frame Only）**，跳过非关键帧（P 帧、B 帧）的解码和追赶过程，以获取 60fps 级别的极致顺滑度。
   * 当用户拖动速度变慢或停止拖拽（例如停滞时间超过 100ms - 150ms）后，自动在后台发起一次**精确定位（Precise Seek）**，解码并呈现绝对精准的高清视频画面。
3. **极低分辨率降采样**：
   * 拖拽预览期间，将解码输出分辨率强行压缩至极低配置（如最大 360p/480p），进一步降低 SwsScale 图像转换及 RGBA 内存带宽的开销。松手后恢复高清晰度。

### 阶段 2：播放驱动与音视频同步优化
1. **音频主时钟同步（Audio Master Sync）**：以声卡实际消耗采样点计算出的绝对时间戳作为同步参考轴，动态控制视频帧的丢弃与延迟展示，消除长时间播放的音视频飘移。
2. **延迟加载 `SeekDecoder`**：在打开媒体文件时仅初始化播放解码器，首个拖拽动作触发时再按需（Lazy Load）打开 `SeekDecoder`。

### 3 阶段 3：视频渲染性能优化（GPU 加速）
1. **GPU 硬件加速渲染（OpenGL / QRhi）**：通过 GPU 纹理直接上传 YUV 原始画面数据，在 Shader 中并行完成 YUV 到 RGB 的色彩转换与拉伸，彻底解放 CPU 算力。
2. **已解码帧缓冲缓存（Decoded Frame Cache）**：内存中设立环形 RGBA/YUV 缓冲区，重用历史拖拽解码结果，实现区间内拖拽 0 延迟响应。
